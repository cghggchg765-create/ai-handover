# AI 交接记录 (AI Handover Records)

> 确保任何 AI 接手后能在 **1 分钟内** 了解项目全貌

## P0 强制触发（v3.1 新增）

🔴 **AI agent 完成任务后必须立即撰写交接记录，这不是"建议"而是 P0 铁律。**

| 规则 | 说明 | 违反后果 |
|------|------|---------|
| 🔴 必须写交接记录 | 完成任务后必须在 `AI交接记录/` 目录创建执行记录 | 视为任务未完成，调用方有权拒收 |
| 🔴 必须实际写文件 | 禁止仅在对话中输出 Markdown 而不写文件 | 视为任务未完成 |
| 🔴 返回结果必须含路径 | 执行方返回给调用方的结果中必须包含交接记录路径引用 | 调用方不得标记任务完成 |
| 🔴 调用方必须验证 | 调用方收到执行结果后必须检查交接是否完整 | 缺一不可，缺失则责令补写 |

**执行方返回结果格式（推荐）**：

```markdown
[任务完成报告]
- **产出物**: [文件路径列表]
- **AI交接记录**: AI交接记录/YYYY-MM-DD_HHmmss_任务简述/执行记录.md
- **关键操作摘要**: [做了什么]
- **遗留问题**: [如有]
```

**调用方验证责任**：

收到执行方返回后，调用方必须检查以下 3 项：
1. ✅ `AI交接记录/` 目录下是否存在对应任务目录
2. ✅ 是否写入了 `执行记录.md` 文件
3. ✅ `AI交接记录/索引.md` 是否已更新

如有缺失，**必须责令执行方补写后再标记任务完成**，不得跳过。

**执行方返回空/失败时的处理**：

```
❌ 调用方违规: 执行方返回空结果 → 调用方自己修改文件
✅ 调用方正确: 执行方返回空结果 → 重新委托/简化任务重试 → 仍失败则升级
```

| 场景 | ❌ 禁止 | ✅ 正确做法 |
|------|---------|-----------|
| 执行方返回空结果 | 调用方自己动手修改 | 重新委托 / 简化任务重试 / 换执行方 |
| 执行方返回错误 | 调用方自己手动修复 | 重新委托 / 换执行方 / 升级处理 |
| 执行方部分完成 | 调用方补齐剩余部分 | 补委托另一个 agent 完成剩余部分 |
| 连续 2 次失败 | 调用方亲自上手 | 换执行方类型 → 仍失败则请求用户介入 |

**🔴 统一失败模式 if-then 表（覆盖所有场景）**：

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 执行方返回空结果 | 重新委托（简化 prompt） | 换执行方类型 |
| 执行方返回错误 | 读取错误信息 → 重新委托 | 换执行方 → 升级 |
| 执行方部分完成 | 补委托另一个 agent | 升级 |
| 连续 2 次失败 | 换执行方类型 | 请求用户介入 |
| 交接记录缺字段 | 责令执行方补写 | 调用方补委托 |
| 索引未更新 | 责令执行方更新 | 调用方补委托 |
| 文件夹命名错误 | 责令执行方重命名 | 调用方补委托 |

## 参数系统

本技能通过参数控制行为，适应不同场景：

| 参数 | 说明 | 默认值 | 可选值 |
|------|------|--------|--------|
| `type` | 记录类型 | handover | handover / progress / decision |
| `project_type` | 项目类型 | auto | auto / software / academic / docs / ops |
| `detail` | 详细程度 | full | full / summary / minimal |
| `output` | 输出格式 | markdown | markdown / json |

参数通过自然语言传递，例如「简单记一下这次修复」→ type=handover, detail=summary。

- **type=handover**：完整交接记录（所有模块）
- **type=progress**：进度更新（仅基本信息 + 产出物 + 下一步）
- **type=decision**：决策记录（仅基本信息 + 技术决策 + 理由）
- **project_type**：影响模板中的字段提示和示例术语
- **detail=summary**：精简版，仅关键字段
- **detail=minimal**：极简版，仅任务名称 + 状态 + 产出

## 智能识别

本技能会自动从自然语言中推断参数，无需用户显式指定：

| 用户表述 | 推断参数 |
|---------|---------|
| "简单记一下" | detail=summary |
| "详细记录" | detail=full |
| "进度更新" | type=progress |
| "决策记录" | type=decision |
| 提到 git/coding | project_type=software |
| 提到 paper/实验 | project_type=academic |
| 提到 deployment/配置 | project_type=ops |

## 模板库

`references/templates/` 目录提供 4 种预设模板（按 project_type 选择）：

| 模板 | 适用 |
|------|------|
| `software.md` | 软件开发 |
| `academic.md` | 学术研究 |
| `docs.md` | 文档写作 |
| `ops.md` | 运维部署 |

## 全局统计

`AI交接记录/统计.md` 自动统计：任务总数、遗留问题分布、按月份分布等。

## 标签系统

每个执行记录可添加标签（如 `#bug #feature #refactor`），用于快速检索。

## 触发规则

本技能在以下任一条件满足时**必须激活**：

| 优先级 | 触发条件 | 说明 |
|--------|----------|------|
| 🔴 P0 强制 | AI agent 完成一个独立任务 | 每个 agent 的每次独立任务都必须写交接记录 |
| 🔴 P0 强制 | 完成一个"用户可感知的独立功能/修复" | ToDoList 中打勾的每一项都是独立交付物 |
| 🔴 P0 强制 | 会话结束前 | 当前对话的所有产出未记录时 |
| 🟡 建议 | 任务中途发现重大问题 | 需要记录遗留问题和决策 |
| 🟡 建议 | 用户说"记录/交接/handover/总结" | 用户显式要求 |

**🔴 多 agent 协作场景的强制流程**：

```
执行方完成任务 → 写 AI交接记录/ → 返回给调用方（含路径引用）
                                          ↓
                        调用方验证交接完整性 → 通过 → 标记任务完成
                                          ↓ (不通过)
                              责令执行方补写后再标记完成
```

## 执行流程

### 第一步：创建时间戳文件夹

```
AI交接记录/YYYY-MM-DD_HHmmss_任务简述/
```

**命名规则：**
- `YYYY-MM-DD`：完成日期
- `HHmmss`：完成时间（24小时制，精确到秒）
- `任务简述`：中文短描述（≤20字），用 `-` 连接多词

**反例（禁止）：**
- ❌ `2026-05-17_部署` — 缺时间、无秒
- ❌ `2026-05-17_fix_bug` — 任务简述应为中文
- ❌ `AI交接记录/deploy/` — 格式完全错误

> 🔴 CHECKPOINT: 文件夹创建后，立即验证命名格式。格式错误 → 重命名后再继续。

### 模块选择矩阵

根据 `type` 和 `detail` 参数选择模块组合：

| 模块 | handover | progress | decision | summary | minimal |
|------|----------|----------|----------|---------|---------|
| 基本信息 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 执行过程 | ✅ | ✅ | ❌ | ✅ | ❌ |
| 产出物 | ✅ | ✅ | ❌ | ✅ | ✅ |
| 技术决策 | ✅ | ❌ | ✅ | ❌ | ❌ |
| 环境信息 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 遗留问题 | ✅ | ✅ | ❌ | ✅ | ❌ |
| 下一步计划 | ✅ | ✅ | ❌ | ✅ | ❌ |

### 第二步：编写执行记录

在文件夹内创建 `执行记录.md`，按模块选择矩阵包含对应字段。

**模板选择**：根据 `project_type` 从 `references/templates/` 选对应模板。

```markdown
# [任务简述]

## 基本信息
- **任务名称**：[一句话描述]
- **执行 AI**：[使用的 AI 模型名称]
- **开始时间**：YYYY-MM-DD HH:mm
- **结束时间**：YYYY-MM-DD HH:mm
- **前置状态**：[项目当前状态、依赖条件、上次进度]
- **标签**：#bug #feature #refactor

## 执行过程
- **关键操作**：
  1. [步骤一] → [结果]
  2. [步骤二] → [结果]
- **遇到的问题**：[如有，描述问题及解决方案]
- **技术决策**：[如有，记录决策及理由]

## 产出物
| 文件/目录 | 类型 | 说明 |
|-----------|------|------|
| `路径` | 文件/目录 | 用途 |

## 后置状态
- **项目当前状态**：[进度百分比或里程碑]
- **环境信息**：
  - 虚拟环境：`[venv 路径]`
  - 依赖变更：[是否安装了新包]
  - 配置变更：[关键配置改动]
- **新增/修改文件数**：[数量]

## 遗留问题
| 序号 | 问题 | 严重程度 | 建议处理方式 |
|------|------|----------|-------------|
| 1 | [问题描述] | 🔴高/🟡中/🟢低 | [建议方案] |

## 下一步计划
- [ ] [计划一]
  - [ ] [计划二]
```

> 🔴 CHECKPOINT: 执行记录写入后，立即检查：
> - 基本信息（任务名称+执行AI+时间）是否完整？
> - 产出物表格是否有内容？
> - 遗留问题是否标记了 🔴🟡🟢？
> 缺字段 → 补充后再继续。

### 第三步：更新索引

在 `AI交接记录/索引.md` 的**文件顶部**（最新记录在上）插入新条目。

**索引格式（强制）：**

```markdown
## YYYY-MM-DD - [任务简述]

| 时间 | 任务 | 产出物 | 状态 |
|------|------|--------|------|
| HH:mm | [子任务描述] | [产出文件/目录] | ✅/🔴/🟡 |
| HH:mm | [子任务描述] | [产出文件/目录] | ✅/🔴/🟡 |

详细信息见: [文件夹名]/

> 📌 当前状态：[一句话描述最新进度]
```

**示例（参考现有格式）：**

```markdown
## 2026-05-17 - Superpowers + GStack 技能部署

| 时间 | 任务 | 状态 |
|------|------|------|
| 2026-05-17 | 部署 obra/superpowers v5.1.0 (14技能) | ✅ 完成 |
| 2026-05-17 | 部署 garrytan/gstack (46技能) | ✅ 完成 |
| 2026-05-17 | AGENTS.md 路由规则更新 (v2.4→v2.5, 4域35场景→5域55场景) | ✅ 完成 |

详细信息见: 2026-05-17_superpowers-gstack部署/

> 📌 60个技能已部署完毕，需重启OpenCode后可用。
```

> 🔴 CHECKPOINT: 索引更新后，立即验证：
> - 新条目是否在索引文件**顶部**？
> - 格式是否与示例一致？
> - 详细信息路径是否指向正确的文件夹？
> 不通过 → 修正后再标记完成。

### 第四步：更新全局统计（可选）

在 `AI交接记录/统计.md` 维护以下统计信息：
- 任务总数
- 遗留问题严重度分布
- 按月份分布

### 第五步：自检

完成后，逐条检查：

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 文件夹命名 | 格式 `YYYY-MM-DD_HHmmss_中文简述`，精确到秒 |
| 2 | 执行记录字段 | 按模块选择矩阵包含对应字段 |
| 3 | 索引已更新 | 新条目出现在索引文件顶部 |
| 4 | 时间准确 | 开始/结束时间精确到分钟 |
| 5 | 遗留问题有严重度 | 每条遗留问题标记了 🔴/🟡/🟢 |
| 6 | 下步计划可执行 | 下一步计划是具体动作，非空泛描述 |
| 7 | 标签完整 | 关键任务已添加 `#xxx` 标签 |

## 项目类型适配

不同项目类型的模板差异：

| 项目类型 | 模板差异 |
|---------|---------|
| software | 默认模板，关注代码变更、架构决策 |
| academic | 强调文献来源、方法选择、实验参数 |
| docs | 强调文档结构、版本变更、审阅记录 |
| ops | 强调配置变更、环境变量、部署步骤 |

**详细模板**：`references/templates/{software,academic,docs,ops}.md`

## 降级策略

当缺失必要信息时，按以下策略处理：

| 缺失信息 | 处理方式 |
|---------|---------|
| Git 变更记录 | 标注「无 Git 信息」，手动描述 |
| 环境配置 | 记录已知部分，未知标注「待补充」 |
| 前置状态 | 若无记录，标注「首次记录」 |
| 开始时间 | 以当前时间作为近似起点 |
| 项目类型 | 默认为 software |

## 特殊场景处理

### 场景 A：初次创建交接体系
如果项目的 `AI交接记录/` 目录不存在：
1. 创建 `AI交接记录/` 目录
2. 创建 `索引.md`（使用上方格式，第一行为标题 `# AI 交接记录索引`）
3. 按正常流程创建第一条执行记录

### 场景 B：项目还未完成
如果任务因中断而未完成：
- 执行记录中的"后置状态"写明中断点
- "遗留问题"中标记中断原因
- "下一步计划"的第一条是"从 [中断点] 继续"

### 场景 C：多项目并存
如果同时维护多个项目：
- 每个项目根目录下维护独立的 `AI交接记录/`
- 在 `说明文档.md` 中列出所有项目的路径
- 切换项目时同时切换对应的交接记录上下文

## 反模式（禁止操作）

| 禁止操作 | 正确替代 |
|----------|----------|
| ❌ 不写执行记录就切换任务 | ✅ 每个独立任务完成后立即记录 |
| ❌ 只更新执行记录不更新索引 | ✅ 两者同步更新，索引在顶部 |
| ❌ 遗留问题不标严重度 | ✅ 每个问题标记 🔴🟡🟢 |
| ❌ "下一步计划"写"继续开发" | ✅ 写具体动作："实现 user API 的 POST 接口" |
| ❌ 文件夹命名缺时间秒 | ✅ 完整 `HHmmss`（可向用户确认） |
| ❌ 无视降级策略导致空字段 | ✅ 标注「待补充」而非留空 |
| ❌ 关键任务无标签 | ✅ 添加 `#bug` `#feature` `#refactor` 等标签 |
| ❌ 仅在对话中输出 Markdown 不写文件 | ✅ 必须实际写入 `AI交接记录/` 目录 |
| ❌ 执行方返回结果不含交接记录路径 | ✅ 返回结果必须包含路径引用 |
| ❌ 调用方收到空/失败结果后自己动手修改 | ✅ 重新委托/简化重试/换执行方 |
