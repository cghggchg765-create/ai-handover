## 📌 为什么需要这个技能？

多 AI 协作中，**信息断层**是最昂贵的隐性成本：

```
会话 A (GPT-4)     →  写了 200 行代码，记录了 3 个决策
会话 B (Claude)    →  不知道 A 做了什么，重新探索了 30 分钟
会话 C (Codex CLI) →  找不到历史记录，重复了 A 的 bug
```

**AI 交接记录**将上下文传递标准化为 **P0 铁律级流程**——让任何 AI 接手时不再"失忆"。

### 解决的核心痛点

| 痛点 | 后果 | 本技能解决 |
|------|------|-----------|
| 不知道有哪些活跃项目 | 重复探索 | `AI交接记录/索引.md` 时间倒序总览 |
| 不知道上次做到哪了 | 进度丢失 | 执行记录含「后置状态」+「下一步计划」|
| 不知道执行记录需要哪些字段 | 格式混乱 | 预设 4 种项目模板 + 模块选择矩阵 |
| 写完记录忘记更新索引 | 记录沉没 | 5 步流程 + 自检清单兜底 |
| 子 agent 返回空结果 | 调用方自己上手（违规） | 集中式失败 if-then 表覆盖所有场景 |

## 🔥 核心设计

### 执行流程（5 步标准化）

```
用户请求完成
     ↓
[Step 1] 创建时间戳文件夹
    AI交接记录/YYYY-MM-DD_HHmmss_任务简述/
     ↓ 🔴 CHECKPOINT：命名格式验证
[Step 2] 按模块选择矩阵编写执行记录
     ↓ 🔴 CHECKPOINT：字段完整性验证
[Step 3] 更新索引.md（最新条目在顶部）
     ↓ 🔴 CHECKPOINT：索引位置+格式验证
[Step 4] 更新全局统计（可选）
     ↓
[Step 5] 自检清单（7 项逐条确认）
     ↓
✅ 交接完成
```

### P0 强制触发体系

本技能在 **4 种场景下必须激活**，违反视为任务未完成：

```mermaid
flowchart LR
    A[独立任务完成] -->|P0 强制| C[写 AI交接记录/]
    B[独立功能/修复] -->|P0 强制| C
    D[会话结束前] -->|P0 强制| C
    E[用户说"记录/交接"] -->|🟡 建议| C
    C --> F[调用方验证 3 项]
    F -->|通过| G[标记完成 ✅]
    F -->|不通过| H[责令补写]
```

**调用方验证 3 项**：
1. ✅ `AI交接记录/` 目录下是否存在对应任务目录
2. ✅ 是否写入了 `执行记录.md` 文件
3. ✅ `AI交接记录/索引.md` 是否已更新

### 参数系统

| 参数 | 说明 | 默认值 | 可选值 |
|------|------|--------|--------|
| `type` | 记录类型 | handover | handover / progress / decision |
| `detail` | 详细程度 | full | full / summary / minimal |
| `project_type` | 项目类型 | auto | auto / software / academic / docs / ops |

**智能识别** — 无需显式指定，从自然语言自动推断：

| 用户表述 | 推断结果 |
|---------|---------|
| "简单记一下" | type=handover, detail=summary |
| "详细记录" | type=handover, detail=full |
| "进度更新" | type=progress |
| "决策记录" | type=decision |
| 提到 paper/实验 | project_type=academic |

## 🛡️ 失败处理矩阵

**集中式 if-then 表** — 覆盖所有异常场景：

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 执行方返回空结果 | 重新委托（简化 prompt） | 换执行方类型 |
| 执行方返回错误 | 读取错误 → 重新委托 | 换执行方 → 升级 |
| 执行方部分完成 | 补委托另一个 agent | 升级 |
| 连续 2 次失败 | 换执行方类型 | 请求用户介入 |
| 交接记录缺字段 | 责令执行方补写 | 调用方补委托 |
| 索引未更新 | 责令执行方更新 | 调用方补委托 |
| 文件夹命名错误 | 责令执行方重命名 | 调用方补委托 |

**红线原则**：

| ❌ 禁止 | ✅ 正确做法 |
|---------|-----------|
| 执行方返回空结果 → 调用方自己修改 | 重新委托 / 简化重试 / 换执行方 |
| 执行方返回错误 → 调用方手动修复 | 重新委托 / 换执行方 / 升级 |
| 调用方收到空结果自己动手 | 重新委托 → 仍失败 → 请求用户介入 |

## 🛡️ 降级策略

| 缺失信息 | 处理方式 |
|---------|---------|

| 前置状态 | 若无记录，标注「首次记录」 |
| 开始时间 | 以当前时间作为近似起点 |
| 项目类型 | 默认为 software |
