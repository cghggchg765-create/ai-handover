# 跨项目模式注册表

> 记录在本项目及历史项目中沉淀的**可复用模式**。新 Agent 在遇到类似场景时应优先参考此表。
> 每条模式记录其首次发现的 Handover-ID，方便追溯上下文。

## 模式清单

| 模式名称 | 描述 | 适用场景 | 首次记录 | Handover-ID |
|----------|------|----------|----------|-------------|
| **错误处理中间件模式** | 使用统一的 `asyncHandler` 包裹所有异步路由处理器，集中错误处理中间件捕获异常，禁止 try/catch 散落各处 | Express/Koa 后端项目；所有异步 API 路由 | 2026-06-25 | `2026-06-25_003_error-middleware` |
| **API 版本控制模式** | URL 路径版本控制（`/api/v1/`、`/api/v2/`），每个版本独立 Router，版本间通过 Adapter 层转换 | 需要长期维护多版本 API 的 Web 服务 | 2026-06-20 | `2026-06-20_api-versioning` |
| **配置管理模式** | 分层配置：`default.yml`（默认）→ `environment.yml`（环境覆盖）→ `local.yml`（本地覆盖，不提交 Git），使用 `convict` 库校验 | 多环境（dev/staging/prod）部署的 Node.js 项目 | 2026-06-18 | `2026-06-18_config-pattern` |
| **仓库模式（Repository）** | 数据访问层统一通过 Repository 接口封装，业务层不直接操作 ORM 或数据库驱动 | 需要切换数据源、或需要 mock 数据层的项目 | 2026-06-15 | `2026-06-15_repo-pattern` |
| **健康检查端点模式** | 所有服务暴露 `GET /health`（存活检查）+ `GET /health/ready`（就绪检查，含各依赖状态） | 需要容器化部署/K8s 编排的任何后端服务 | 2026-06-10 | `2026-06-10_health-check` |

## 使用规则

1. 发现新模式 → 追加到本表末尾（按时间序）
2. 模式必须提供至少 1 个 Handover-ID 作为证据
3. 如果有更优模式取代现有模式，标记为 `⚠️ 已弃用` 并在描述中说明替代
