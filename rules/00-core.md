## 1. Overview

本文件定义 ai-handover 的 **9 条 IRON RULES**（铁律）。这些规则对所有 agent 具有强制约束力，**不可协商**。违反任何一条 → 调用方必须拒绝任务输出，责令修正后方可验收。

每条铁律包含：规则描述、违反后果、以及指向详细规则文件的引用。

---

## 2. IRON RULE #1–#9

### IRON RULE #1 — 强制写交接记录

| 项 | 内容 |
|---|------|
| **描述** | 每次任务完成后，子 agent **必须** 在 `AI交接记录/` 目录下创建执行记录文件，否则视为任务未完成 |
| **违反后果** | 调用方拒收返回结果，责令补写 |
| **引用** | `rules/01-handover.md §2` |

### IRON RULE #2 — 模板格式强制

| 项 | 内容 |
|---|------|
| **描述** | 交接记录必须使用 `rules/01-handover.md §3` 定义的统一模板，禁止自定义格式 |
| **违反后果** | 调用方拒绝，责令按模板重写 |
| **引用** | `rules/01-handover.md §3` |

### IRON RULE #3 — Git trailers 强制

| 项 | 内容 |
|---|------|
| **描述** | 涉及 git 操作的交接记录必须包含标准 `Handover:`、`Signed-off-by:`、`AI:` 等 trailer 行 |
| **违反后果** | 调用方拒绝，责令补全 |
| **引用** | `rules/03-git.md §2` |

### IRON RULE #4 — 交接链强制

| 项 | 内容 |
|---|------|
| **描述** | 每个交接记录必须在 frontmatter `handover_chain` 字段中引用上一个交接记录的路径，形成可追溯链。首个交接填 `null` |
| **违反后果** | 链断裂视为无效交接，调用方拒绝 |
| **引用** | `rules/01-handover.md §4` |

### IRON RULE #5 — 串行状态门控

| 项 | 内容 |
|---|------|
| **描述** | `todo.md` 中同一时刻只能有一个任务处于 `in_progress` 状态。不得跳过前一状态直接标记后续任务 |
| **违反后果** | 调用方要求修正状态后重审 |
| **引用** | `rules/02-coordination.md §2` |

### IRON RULE #6 — 并行文件锁

| 项 | 内容 |
|---|------|
| **描述** | 多 agent 并行时，每个 agent 写入前必须检查 `.locks/` 目录下的文件锁，冲突时须等待或协调 |
| **违反后果** | 写入冲突文件 → 调用方撤销写入并责令排队 |
| **引用** | `rules/02-coordination.md §3` |

### IRON RULE #7 — hot.md 强制更新

| 项 | 内容 |
|---|------|
| **描述** | 每个任务完成后必须更新 `hot.md` 中的工作目录、当前状态、下一步提示等信息 |
| **违反后果** | 调用方拒收，责令更新后再审 |
| **引用** | `rules/05-memory.md §2` |

### IRON RULE #8 — 新 Agent 入职流程

| 项 | 内容 |
|---|------|
| **描述** | 首次接入本系统的 agent，必须先执行 `rules/05-memory.md §5` 定义的入职流程，读取历史交接记录并完成环境验证 |
| **违反后果** | 跳过入职 → 调用方禁止其执行正式任务 |
| **引用** | `rules/05-memory.md §5` |

### IRON RULE #9 — 分支策略强制

| 项 | 内容 |
|---|------|
| **描述** | 所有代码/文档变更必须遵循 `rules/03-git.md §1` 定义的分支策略（如 `main` 只读、`develop` 合入门、`feature/*` 开发分支） |
| **违反后果** | 调用方阻止合并，责令重走分支策略 |
| **引用** | `rules/03-git.md §1` |

## 3. 调用方拒绝协议

调用方（主 agent / build）在收到子 agent 返回结果后，按以下流程执行 **5 项拒绝检查**：

```
接收子 agent 返回结果
  │
  ├─ ① 交接记录是否存在？
  │    └─ NO → 拒绝，责令创建
  │
  ├─ ② 模板格式正确？
  │    └─ NO → 拒绝，责令重写
  │
  ├─ ③ 必填字段完整？（如 handover_chain、hot.md 更新）
  │    └─ NO → 拒绝，责令补全
  │
  ├─ ④ 状态门控合法？
  │    └─ NO → 拒绝，责令修正
  │
  └─ ⑤ Git trailors 完备（如涉及 git 操作）？
       └─ NO → 拒绝，责令补全

全部通过 → 标记任务完成 ✅
任何一项不通过 → 拒绝 ❌
```

**关键原则**：
- 拒绝后调用方应以 `[REJECT]` 前缀明确告知子 agent 哪项检查未通过
- 子 agent 修正后重新提交，调用方重新执行全部 5 项检查
- 连续 2 次拒绝 → 升级至 `[ESCALATE]` 协议

## 4. 统一失败模式 if-then 表

| # | 失败模式 | 一线修复 | 兜底方案 |
|---|---------|---------|---------|
| 1 | **空结果**：子 agent 返回 null/空/无实质内容 | 重试 1 次 → 简化 prompt | 换 agent 类型 → [ESCALATE] |
| 2 | **错误**：子 agent 返回 ERROR/EXCEPTION | 委托 researcher 诊断错误 | 记录交接 → [ESCALATE] |
| 3 | **部分完成**：子 agent 仅完成部分子任务 | 补委托另一个 agent 完成剩余部分 | 输出已完成/未完成清单 |
| 4 | **连续失败**：同一子 agent 连续 2 次失败 | 换 agent 类型 | 请求用户介入 |
| 5 | **缺字段**：交接记录 frontmatter 必填字段缺失 | 责令子 agent 按模板补全 | 调用方写错误记录 → [ESCALATE] |
| 6 | **索引未更新**：`AI交接记录/索引.md` 未同步更新 | 责令子 agent 补写索引 | 调用方手动触发索引重建 |
| 7 | **命名错误**：交接记录文件名不符合 `YYYY-MM-DD_HHmmss_任务简述` 规范 | 责令子 agent 重命名 | 记录错误 → [ESCALATE] |
| 8 | **状态跳转非法**：`todo.md` 跳过未完成状态直接标记后续 | 责令子 agent 修正状态 | 调用方重置 todo → [ESCALATE] |
| 9 | **队列堵塞**：`.locks/` 锁未释放导致其他 agent 无法写入 | 清理残留锁文件 | 通知所有 agent 重置锁状态 |

## 5. 验证责任矩阵

### 调用方（主 agent / build）验证项

| # | 验证内容 | 对应铁律 | 验证方式 |
|---|---------|---------|---------|
| 1 | 交接记录目录是否存在且已写入 | IRON #1 | 检查 `AI交接记录/` 路径 |
| 2 | 模板格式符合规范 | IRON #2 | 对比模板结构 |
| 3 | frontmatter 必填字段完整 | IRON #4 | 检查 `handover_chain` 等字段 |
| 4 | 状态门控合法（串行约束） | IRON #5 | 检查 `todo.md` 状态 |
| 5 | Git trailers 完备（如适用） | IRON #3 | 检查 commit message |

### validate.sh 验证项

| # | 验证内容 | 对应铁律 | 验证方式 |
|---|---------|---------|---------|
| 1 | 所有 `AI交接记录/` 文件 YAML 格式正确 | IRON #2 | `yq` / 解析器校验 |
| 2 | `handover_chain` 链完整性（无断链） | IRON #4 | 逐文件追溯验证 |
| 3 | 交接记录文件名格式正确 | IRON #1 | 正则匹配 `^\d{4}-\d{2}-\d{2}_\d{6}_` |
| 4 | `hot.md` 最后更新时间与最新交接记录一致 | IRON #7 | 时间戳对比 |
| 5 | `todo.md` 无死锁状态（多个 `in_progress`） | IRON #5 | 状态机检查 |
| 6 | `.locks/` 目录无残留锁 | IRON #6 | 检查锁超时 |
| 7 | 入职流程完成标记 | IRON #8 | 检查 `.onboarding` 标记文件 |
| 8 | 分支名称符合策略规范 | IRON #9 | 正则匹配 `^(feature\|fix\|docs)/` |
